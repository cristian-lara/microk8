# Tabla de aprendizaje (orquestador)

Errores comunes, causas y soluciones que el orquestador y los flujos pueden reutilizar al levantar o modificar servicios. Actualizar cuando un patrón se repita o sea útil para otros servicios.

| Servicio / Área | Error o síntoma | Causa | Solución | Ref |
|-----------------|-----------------|--------|----------|-----|
| NFS / Storage   | `showmount -e` no muestra exports | NFS no habilitado o permisos no configurados en Synology | Activar NFS en File Services; carpeta compartida con NFS permissions para `192.168.50.0/24`; en VM `apt install nfs-common` | `docs/08-notas-implementacion.md` §2 |
| NFS / Storage   | PVC Pending, provisioner no crea volumen | StorageClass por defecto incorrecta o hostpath en uso | Usar `nfs-storage` como default; deshabilitar hostpath-storage en prod | `docs/08-notas-implementacion.md` §3 |
| Cloudflare      | "No DNS record found for this domain" (Access) | Falta registro DNS para el hostname | Crear registro Tunnel/CNAME para el subdominio en DNS de cld-lf.com | `docs/08-notas-implementacion.md` §4.2 |
| Cloudflare      | NXDOMAIN al abrir subdominio | Mismo que arriba | Crear CNAME/Tunnel y esperar propagación | `docs/08-notas-implementacion.md` §4.3 |
| YAML / Manifests| Rechazo por reglas de prod | Imagen `:latest`, sin resources/probes/securityContext, secretos en claro | Aplicar `.cursor/rules/k8s-yaml-prod.mdc`: versión fija, resources, probes, securityContext, Vault/ExternalSecret | `k8s-yaml-prod.mdc` |
| Vault / PostgreSQL | Vault no puede crear usuarios DB | Usuario `vault` en PostgreSQL sin CREATEROLE o sin permisos en la DB | Ejecutar `create-vault-db-user.sh` y `grant-vault-to-gitea.sh` (o equivalente) en el orden indicado | `docs/k8s/vault/vault-postgres-integration.md` |
| General         | Paso marcado OK pero otro agente no ve cambios | No se hizo pull tras el step | Hacer pull (y push si aplica) al completar cada step según `workflow/RULES.md` | `workflow/RULES.md` |
| Ingress         | Comportamiento distinto o se sugiere otro addon | Uso del addon `microk8s enable ingress` | Instalar Ingress vía **Helm** (chart ingress-nginx); no usar addon. Ver `docs/02-microk8s-bootstrap.md` y `docs/08-notas-implementacion.md` §3b. | `docs/08-notas-implementacion.md` §3b |
| Redeploy / datos | Pérdida de datos al re-aplicar o tras apagado | Borrado de PVCs o uso de storage no persistente | Usar solo **apply** / **helm upgrade** (no uninstall). Datos en PVCs con `nfs-storage` (NFS). Tras reboot, Vault puede requerir **unseal**. Ver `docs/08-notas-implementacion.md` §10. | `docs/08-notas-implementacion.md` §10 |
| Webhooks / URLs | Webhooks o callbacks no funcionan desde fuera; app configurada con localhost | Variables de entorno del deploy (ROOT_URL, WEBHOOK_URL, etc.) con `localhost`, `127.0.0.1` o IP privada | Siempre usar dominio real: `https://<subdominio>.cld-lf.com` (dominio base **cld-lf.com**). Investigar si la app tiene webhooks; configurar con subdominio. Auditor rechaza localhost. | `workflow/skills/webhooks-and-public-urls.md`, `workflow/audit/checklist-production.md` §6 |
| MicroK8s / cloudnative-pg | "Addon cloudnative-pg was not found in any repository" | El addon está en el repositorio **community**, no en el core | Ejecutar primero `microk8s enable community`, luego `microk8s enable cloudnative-pg` | `docs/08-notas-implementacion.md` §9 |
| MicroK8s / cloudnative-pg | "Apply failed with conflicts: conflicts with helm" | CloudNativePG ya instalado vía Helm; addon y Helm gestionan los mismos CRDs/webhooks | Si el operador ya corre: usar esa instalación y desplegar el cluster; no hace falta el addon. Si se quiere solo addon: `helm uninstall <release> -n <ns>` y luego `microk8s enable cloudnative-pg` | `docs/08-notas-implementacion.md` §9 |
| CloudNativePG / Helm | `ConfigMap "cnpg-default-monitoring" exists and cannot be imported: invalid ownership metadata`; missing `app.kubernetes.io/managed-by`, `meta.helm.sh/release-name` | Recursos en `cnpg-system` creados por addon o instalación manual anterior; no tienen etiquetas/anotaciones de Helm | Instalación limpia con Helm: borrar el namespace y reinstalar. En la VM: `microk8s kubectl delete namespace cnpg-system`. Luego re-ejecutar `./docs/k8s/postgres/install-cnpg-operator.sh`. **No** borrar `cnpg-system` si ya tienes clusters PostgreSQL (ej. `platform-db`) en uso y gestionados por ese operador. | `docs/08-notas-implementacion.md` §9 |
| CloudNativePG / Helm | CRD `backups.postgresql.cnpg.io` (u otras) exists; `meta.helm.sh/release-namespace` must equal "cnpg-system", current value is "platform" | Las CRDs fueron instaladas por una release de Helm en otro namespace (p. ej. `platform`); la nueva instalación en `cnpg-system` no puede adoptarlas | **Opción A (instalación limpia, sin clusters que conservar):** desinstalar release en platform: `microk8s helm3 list -n platform` → `microk8s helm3 uninstall <release> -n platform`. Borrar CRDs: `microk8s kubectl get crd -o name \| grep postgresql.cnpg.io \| xargs microk8s kubectl delete`. Re-ejecutar script de instalación. **Opción B (conservar clusters):** instalar sin crear CRDs: `microk8s helm3 install cnpg cnpg/cloudnative-pg -n cnpg-system --create-namespace --set crds.create=false`. Ver `docs/08-notas-implementacion.md` §9. | `docs/08-notas-implementacion.md` §9 |
| CloudNativePG / NFS | initdb falla: "FATAL: data directory ... has wrong ownership"; HINT: server must be started by the user that owns the data directory | NFS con squash "Map all users to admin": archivos en NAS con UID del admin (ej. 1024), contenedor PostgreSQL corre como 26 | **Recomendado:** Synology NFS Permissions → Squash **No mapping**. Luego **borrar el Cluster** (no solo PVC/Job) para salir de "unrecoverable": `microk8s kubectl delete cluster platform-db -n platform`; después `./docs/k8s/postgres/apply-postgres-platform.sh`. **Alternativa:** mantener squash y en Cluster `podSecurityContext` usar `runAsUser: 1024`, `runAsGroup: 100`, `fsGroup: 100` (UID/GID admin Synology). Ver `docs/08-notas-implementacion.md` §2. | `docs/08-notas-implementacion.md` §2 |
| CloudNativePG | Cluster "unrecoverable and needs manual intervention"; apply dice "unchanged"; no hay pods | Tras borrar PVC o fallos de initdb, el operador marca el cluster como unrecoverable y no vuelve a crear recursos solo con apply | Borrar el recurso Cluster y re-aplicar: `microk8s kubectl delete cluster platform-db -n platform`. Luego `./docs/k8s/postgres/apply-postgres-platform.sh`. Se creará nuevo PVC y Job initdb. | `docs/08-notas-implementacion.md` §2 |
| Bitnami / Helm | `ImagePullBackOff`, `unexpected media type text/html`; imagen Bitnami no descarga | Desde agosto 2025, Bitnami requiere suscripción para la mayoría de imágenes. Docker Hub devuelve HTML en lugar de imagen | Usar imágenes oficiales en lugar de Bitnami. Ej: `quay.io/keycloak/keycloak` en vez de `bitnami/keycloak`. Cambiar chart si es necesario (codecentric/keycloakx para Keycloak). | `docs/k8s/keycloak/` |
| Keycloak / codecentric chart | `Invalid value for option 'kc.cache-stack': jdbc-ping` o vacío | El chart usa `cache.stack: default` → `jdbc-ping`, no soportado en Keycloak 24+. `custom` genera valor vacío, también inválido. Valores válidos: tcp, udp, kubernetes, ec2, azure, google. | Usar `cache.stack: kubernetes` en values.yaml. Para single-replica funciona bien (no hay peers que descubrir). | `docs/k8s/keycloak/values-keycloak-prod.yaml` |
| Cloudflare Tunnel | Error 502 Bad Gateway; "Unable to reach the origin service" | `cloudflared` corre en la VM (fuera del cluster) y no puede resolver DNS internos de K8s (`.svc.cluster.local`) | En Public Hostname usar `localhost:80` (apunta al Ingress controller) en vez de `service.namespace.svc.cluster.local`. El Ingress routea al servicio correcto. | `docs/08-notas-implementacion.md` §12 |
| PostgreSQL / Apps | `password authentication failed for user "X"` al conectar desde app | Usuario existía de intento anterior con password diferente; el Secret tiene password nuevo pero PostgreSQL tiene el viejo | Actualizar password en PostgreSQL: `ALTER USER X WITH PASSWORD 'nuevo-password';` ejecutado via `kubectl exec`. | `docs/08-notas-implementacion.md` §12 |
| Gitea / Helm | `Init:CrashLoopBackOff`; chart despliega postgresql-ha y valkey-cluster internos | El chart de Gitea por defecto habilita subcharts de DB y cache que duplican infraestructura | Deshabilitar subcharts en values: `postgresql-ha.enabled: false`, `valkey-cluster.enabled: false`, `redis-cluster.enabled: false`. Configurar `gitea.config.database` con DB externa. | `docs/k8s/gitea/values-gitea-prod.yaml` |
| Helm values | Errores continuos en values.yaml; demasiadas iteraciones prueba-error | No se validó el values antes de aplicar; se copian defaults sin entenderlos; no se consulta documentación del chart | **Obligatorio:** 1) `helm show values <chart>` antes de escribir; 2) leer README del chart; 3) `helm lint` y `helm template --debug` antes de aplicar; 4) `--dry-run` antes de install/upgrade. **Ante error:** ejecutar protocolo de error completo (capturar logs, identificar causa raíz, re-validar). Ver regla `.cursor/rules/helm-values-strict.mdc` y checklist §9. | `.cursor/rules/helm-values-strict.mdc`, `workflow/audit/checklist-production.md` §9 |
